# Boreal Chessmaster: Architecture & Engine Documentation

## Overview
The Boreal Chessmaster platform operates using a **Hybrid AI Architecture** that combines the deterministic reliability of a heuristic engine with the dynamic adaptability of a Neural Network. 

Many users assume the Heuristic Engine (MCTS) is solely used for offline data generation to train the Neural Network. **This is incorrect.** Both the Neural Network and the Heuristic Engine work together in real-time during live inference.

Here is the exact breakdown of how the system processes a live threat scenario.

---

## 1. The Frontend (viz_engine.js)
The frontend acts as the sensory layer. It tracks the physical location, velocity, and trajectory of all threats in the theater. 
* **Temporal Data:** The frontend stores the rolling path history of threats.
* **Uplink:** Every few seconds, the frontend takes the *current* coordinates of all threats (not their destinations) and sends them to the backend `/evaluate_advanced` API.

---

## 2. The Neural Network (RL)
*Location: `core/engine.py -> extract_rl_features`*

When the backend receives the live state, it first passes the data to the Neural Network.
1. The backend extracts an **18-Dimensional Tactical Feature Vector** for PPO / neural inference.
2. This vector includes temporal/trajectory awareness data (e.g., `has_marv`, `has_mirv`, `total_mirv_warheads`), ammo stress, and directional geometry.
3. A separate **3-D Temporal Context** (`total_assigned`, `assigned_ratio`, `high_threat_unassigned`) is passed only to the MCTS layer so strategic rollouts can reason about commitment and re-engagement without changing the PPO checkpoint shape.
4. **The NN Output:** The Neural Network **does not directly assign interceptors**. Instead, it outputs `doctrine_weights`. These weights represent a high-level strategic posture (e.g., "Prioritize PAC-3s right now" or "Save Meteor for later").

---

## 3. The Heuristic Engine (TacticalEngine)
*Location: `core/engine.py -> TacticalEngine.get_optimal_assignments`*

Once the NN outputs the `doctrine_weights`, they are fed directly into the **Heuristic Engine**. 
1. The Heuristic Engine calculates a mathematical "Utility Score" for every possible Base-Weapon-Threat pairing.
2. It uses hard physics: calculating Time-To-Impact, maximum interceptor ranges, and base Probability of Kill (Pk).
3. It multiplies these baseline physics scores by the `doctrine_weights` provided by the Neural Network.
4. **The Output:** The Heuristic Engine generates the final `tactical_assignments` (e.g., "Fire PAC-3 from Base A at Threat 1").

**Why do it this way?** By forcing the NN to operate through the Heuristic Engine, the system guarantees 100% physical validity. The NN cannot accidentally order a base to fire a missile it doesn't have, or fire at a target out of range, because the Heuristic Engine acts as an unbreakable physics mask.

---

## 4. Monte Carlo Tree Search (MCTS Rollouts)
*Location: `core/engine.py -> StrategicMCTS.run_mcts_rollout`*

**MCTS is used in BOTH Data Generation and Live Inference.**

Once the `tactical_assignments` are generated by the Heuristic Engine, the system needs to know how confident it should be in this plan.
1. **Live Inference:** Before returning the assignments to the frontend, the backend runs 50 fast stochastic MCTS rollouts of the exact engagement plan. It rolls virtual dice against the Pk tables (accounting for weather, MARV penalties, and the 3-D temporal context) to simulate the outcome 50 times. The average survival rate is returned as the **"Confidence Score"** displayed on the UI.
2. **Data Generation:** During training mode, this exact same MCTS logic is run for thousands of iterations across randomized plans to find the absolute mathematically optimal assignment. This "Ground Truth" plan is then used to train the Neural Network, while the temporal context is still retained for strategic scoring.

---

## 5. Model Execution Differences (CORTEX vs PPO)
The behavior described above applies to the standard models (e.g., **CORTEX-1** and the base **Heuristic** mode).
However, if the user selects the **PPO Direct Action** model (which was introduced to maximize low-latency performance):
1. The **Heuristic Engine is BYPASSED** for tactical assignments.
2. The PPO model evaluates the state and outputs a direct **Action Mask** (explicitly mapping interceptors to threats without the heuristic intermediary).
3. The assignments are still passed to **MCTS**, which performs the stochastic rollouts using the same 3-D temporal context purely to calculate the Confidence Score for the UI.

---

## 6. Salvo Logic (Multiple Firings from Temporality)
Because temporality and maneuvering reduce the Probability of Kill (Pk), the Heuristic engine will dynamically compensate by issuing **Salvo** orders (multiple firings at the same target).
* **Example:** If a standard Ballistic missile has a 95% Pk with a PAC-3, the engine fires **one** interceptor.
---

## 8. Synchronized Sinusoidal Physics (Unified Truth)
To ensure that all views (Strategic, 3D Sandbox, and Kinetic Chase) provide a consistent tactical picture, the system utilizes a **Unified Sinusoidal Jink Model**:

*   **Source of Truth:** The `viz_engine.js` (Dashboard) acts as the Master Physics Engine. It calculates MARV evasive maneuvers using a persistent sinusoidal state (`jinkPhase` and `jinkV`).
*   **Physics Constants:** The model is tuned to match the backend `simulate_interception.py` physics:
    *   **Oscillation Period:** ~3.8 seconds (visualized as smooth S-curves).
    *   **Lateral Amplitude:** 750 m/s (modeled as `jinkMag` in the simulation loop).
*   **Master-Follower Sync:** The dashboard broadcasts a `STATE_SYNC` packet via `BroadcastChannel` in every frame. Secondary views like `kinetic_3d.html` operate as **Surrogates**, disabling their local physics and snapping their meshes to the broadcasted X, Y, Z coordinates. This guarantees that an "Intercept Failure" on the dashboard is visually identical in the 3D sandbox.

---

## 7. Summary of Temporality & System Power
Adding temporality fundamentally increases the **combat power** of the Boreal Chessmaster system through four key advantages:

1.  **Predictive Salvo Management:** By understanding the temporal window *before* a MIRV separates or a MARV begins its jink, the engine can authorize "Overwhelming Fire" (Salvo) to destroy the target while its Pk is still high, saving dozens of missiles that would have been wasted in a terminal-phase chaos.
2.  **Kinetic Fidelity (Physics):** Temporality allows us to use **Proportional Navigation** instead of simple straight-line tracking. This allows the AI to calculate the most energy-efficient curved path to intercept, accounting for the interceptor's boost phase and the target's evasive history.
3.  **Dynamic Re-engagement:** Because the system tracks state over time, it can detect when an in-flight missile has failed or when a threat's behavior has worsened (e.g., it started jinking), allowing for an immediate "Follow-on Shot" without waiting for a manual operator cycle.
4.  **Explainable AI (XAI):** For a human commander, a static dot on a map is a mystery. A **temporal trail** is a story. It shows where the threat came from, how fast it is moving, and why the AI decided to engage it at that specific geographic coordinate. This builds the trust necessary for high-stakes autonomous defense.
